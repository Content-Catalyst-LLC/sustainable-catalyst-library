# Review Gate Workflow — v3.5.0

## Draft

Use while the research question, methods, Sources, Claims, and project structure are still being developed.

## Internal review

Use when the project is coherent enough for internal inspection but is not yet ready for formal quality review.

Suggested activities:

- source review;
- methodology review;
- evidence-gap identification;
- provenance review;
- preliminary issue creation.

## Quality review

Use for formal domain reviews.

Record review outcomes separately for methodology, evidence, integrity, ethics, privacy, legal, accessibility, publication, reproducibility, and cross-product readiness as applicable.

## Conditionally approved

Use when the project is acceptable only if recorded actions are completed.

Conditional approval should have:

- explicit review findings;
- required actions;
- issue due dates;
- exception rationale where applicable.

## Approved

The plugin requires:

- a score of at least 70;
- no open critical issues;
- no failed reviews.

Approval remains a human governance decision.

## Published

The same minimum controls apply as Approved.

Before publication, also review:

- public visibility;
- copyright and licensing;
- privacy;
- accessibility;
- transparency summary;
- Source-integrity warnings;
- export contents;
- public pathway and semantic navigation.

## Archived

Use when active work or publication has ended.

Archiving does not delete reviews, issues, policies, approvals, or provenance history.

## Gate transition history

Every transition records:

```text
event ID
previous gate
new gate
quality score
decision note
user
timestamp
```

## Exceptions

An exception should contain:

- the underlying issue;
- why the control cannot be satisfied;
- risk acceptance rationale;
- approver;
- expiry date;
- compensating control.

An exception does not automatically resolve or close the issue.
