# Source Integrity Workflow — Knowledge Library v3.1.0

## 1. Preserve the original Source

Do not overwrite an old Source record with a replacement edition, correction, or retraction notice.

Create a separate Research Source when the replacement is independently citable.

## 2. Record version identity

Record the version label, version number, release date, and version-family root.

Use a version relationship to describe how the new Source relates to the earlier Source.

## 3. Set the integrity status

Choose the status that reflects the reviewed record.

An incoming Supersedes, Corrects, Replaces, or Retracts relationship may generate a suggested status, but the platform does not apply it automatically.

## 4. Record the notice

Add:

- notice date
- official notice URL
- reviewer explanation
- public-notice setting
- recommended replacement

Use the official publisher, issuing institution, standards body, repository, or regulator notice where possible.

## 5. Review the impact report

Check connected:

- projects
- documents
- Evidence Notes
- Claims

The impact report identifies review obligations. It does not determine whether every dependent claim is invalid.

## 6. Review project decisions

For every affected project, record whether the Source was reviewed, replaced, retained for critical context, or excluded.

## 7. Preserve evidence history

Do not automatically delete quotations or Evidence Notes from corrected or retracted Sources.

Review:

- whether the exact passage was affected;
- whether the replacement preserves or changes the claim;
- whether the Source is retained as historical evidence;
- whether the claim should be revised, disputed, or retired.

## 8. Publish the notice

Enable the public notice only after the integrity record and replacement guidance have been reviewed.

Critical statuses remain publicly visible so the historical record is not erased.

## 9. Run the integrity scan

Open:

```text
SC Library → Source Integrity
```

Run scan batches until Complete.

The scan rebuilds incoming relationships, detects status conflicts, refreshes Source hashes, and regenerates impact reports.

## 10. Recheck exports

After replacing a Source in a project, regenerate the bibliography and project evidence packet.
